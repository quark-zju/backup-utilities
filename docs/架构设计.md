# 备份工具架构设计（高层）

## 1. 目标与范围

目标：把多种来源的“备份单元”增量备份到本地目录，产物适合长期离线存储（如光盘/移动硬盘），并支持可选加密。

支持来源（协议）：
- GitHub（repo 级别）
- Google Drive（目录级别）
- 本地路径（目录级别）
- 局域网路径（如 SMB 挂载后路径）
- SSH 可达路径（远端目录级别）

非目标（第一阶段）：
- 分布式对象存储去重（跨单元块级去重）
- 实时同步
- 全自动密钥托管

---

## 2. 核心概念

### 2.1 备份单元（Backup Unit）
一个可独立判断“是否变化”、可独立产出备份文件的最小对象。
示例：
- `github/<owner>/<repo>`
- `gdrive/<drive_or_user>/<folder_id>`
- `local/<name>`（映射到一个本地目录）

要求：
- 每个单元在目标目录内使用 O(1) 个主文件（建议 2~4 个）：
  - 数据文件：`payload.tar.zst` 或 `payload.tar.zst.enc`
  - 元数据：`metadata.json`
  - 可选：`history.log`（追加式简短审计日志）
- 可单独恢复，不依赖其他单元。

### 2.2 元数据（metadata）
建议分为两层：
- 通用字段（所有协议都要有）
- 协议字段（GitHub/GDrive/SSH 各自定义）

通用字段建议：
- `unit_id`：稳定标识（如 `github/user/repo`）
- `protocol`：`github|gdrive|local|ssh|...`
- `snapshot_time`：本次备份完成时间（UTC）
- `source_fingerprint`：协议输出的“变化指纹”（字符串或结构化）
- `payload`：
  - `path`
  - `size_bytes`
  - `sha256`
  - `compressed`（zstd 级别等）
  - `encrypted`（bool）
- `tool_version`

协议字段示例（GitHub）：
- `default_branch`
- `refs`（`ref -> commit_hash` 映射，可裁剪）
- `pushed_at`

### 2.3 备份配置（Backup Config）
放在目标备份目录内，不含凭据。
建议路径：`<backup_root>/config/backup_config.toml`

配置包含：
- 全局设置：并发数、压缩级别、默认加密策略
- 单元选择：
  - 显式包含（include）
  - 显式排除（exclude）
- 协议配置：如 GitHub 用户/组织列表、是否包含 fork、是否包含私有仓库
- 加密策略：
  - 默认规则（例如私有 repo 默认加密）
  - 单元级覆盖

凭据来源：环境变量 / 系统 credential manager / 外部 CLI 登录态（如 `gh auth`）。

## 3. 目录布局建议

```text
<backup_root>/
  config/
    backup_config.toml
  units/
    github/<owner>/<repo>/
      metadata.json
      payload.tar.zst(.enc)
      history.log                # optional
    gdrive/<account>/<folder_id>/
      metadata.json
      payload.tar.zst(.enc)
  state/
    index.json                   # unit_id -> 最新状态摘要（可重建）
  logs/
    run-YYYYmmdd-HHMMSS.log
```

说明：
- `state/index.json` 是加速索引，不是唯一真相（可由 units 目录重建）。
- 恢复时以每个单元目录为边界，降低损坏扩散。

## 4. 增量策略（建议）

先实现“快照级增量”（推荐第一阶段）：
- 不做块级差分。
- 每次只在“变化检测为真”时重打包该单元。
- 未变化单元直接跳过。

优点：
- 实现复杂度低，可靠性高。
- 产物简单，易离线保存与恢复。

后续可扩展（第二阶段）：
- 单元内差分（如基于 tar 清单 + rolling hash）
- 或保留多版本快照（按策略清理）

---

## 5. 加密设计（现代实践）

你提到的 `AES-GCM + scrypt` 是合理路线。建议细化为：

- KDF：`scrypt(N=2^15, r=8, p=1)` 起步，参数写入文件头，便于未来升级。
- 数据加密：`AES-256-GCM`。
- 每个备份单元每次备份使用独立随机 salt + nonce。
- 加密范围：仅 payload（metadata 默认明文，便于快速判断是否需要恢复）。
- 完整性：GCM tag + metadata 内记录密文 sha256。
- 密码输入：运行时输入，不落盘；可选从 `BACKUP_PASSPHRASE` 环境变量读取。

密文封装建议：
- `payload.tar.zst.enc` 文件头包含：版本、kdf 参数、salt、nonce、aad 摘要。
- AAD 绑定 `unit_id + snapshot_time + payload_path`，防止跨单元替换攻击。

依赖建议：
- 首选 `cryptography`（你已经指定，合理）。

## 6. 协议抽象接口

建议 Python 抽象（伪接口）：

- `discover_units()`：可选，列举候选单元
- `compute_fingerprint(unit, prev_metadata)`：输出变化指纹
- `needs_backup(prev_metadata, new_fingerprint)`：判断是否需要备份
- `export_snapshot(unit, staging_dir, prev_metadata)`：导出本次快照到 staging（目录或文件）
- `collect_protocol_metadata(unit)`：协议特有元数据

通用编排器负责：
- 调用协议层
- 打包 tar.zst
- 可选加密
- 原子写入（临时文件 + rename）
- 更新 metadata / index / 日志

## 7. GitHub 协议（第一阶段）建议

优先使用 `gh` CLI（减少 Python 第三方依赖）：
- 依赖前提：用户已 `gh auth login`
- 列举 repo：`gh repo list`
- 获取 refs / pushed_at：`gh api`
- 拉取数据：
  - 方案 A：`git clone --mirror` 到临时目录后打包（最稳）
  - 方案 B：`gh repo archive`（简单但 refs 信息较弱）

建议第一版采用方案 A（镜像仓库），因为：
- refs 完整，利于增量判断和未来恢复。
- 裸仓库打包体积通常可接受。

GitHub 单元变化指纹建议：
- `fingerprint = sha256(sorted(ref -> hash))`
- 可辅以 `pushed_at` 做快速短路。

默认策略建议：
- 非 fork 默认选中
- 私有 repo 默认加密
- 可在配置覆盖

## 8. TUI 与 CLI

### 8.1 CLI（先做）
先把可脚本化能力打牢，再包 TUI。

建议子命令：
- `backup init --root <path>`
- `backup discover github --user <u>`
- `backup select add|remove <unit_id>`
- `backup run [--unit <id>] [--dry-run]`
- `backup status`
- `backup verify [--unit <id>]`

### 8.2 TUI（第二步）
可用 `whiptail` 封装：
- 协议选择
- 单元勾选
- 加密策略设定
- 运行备份/查看日志

实践建议：
- TUI 本质调用 CLI 子命令，避免双份业务逻辑。

## 9. 一致性与容错

- 原子落盘：
  - 先写 `*.tmp`，校验 hash 后 rename
- 断点容错：
  - 启动时清理/识别残留 tmp
- 并发控制：
  - 单元级锁文件（防止同单元并发备份）
- 日志：
  - 运行日志 + 单元历史日志
- 可验证性：
  - `backup verify [--unit]` 校验 metadata 与 payload hash

---

## 10. 已确认约束

1. 历史版本策略：
- 每个备份单元只保留最新一份快照（覆盖更新），不保留 N 份历史。

2. metadata 可见性：
- metadata 明文可接受，允许暴露 repo/path 等标识信息。

3. GitHub 备份粒度：
- 仅备份 git bare repo mirror（含所有 branch/refs）。
- 不包含 issues、releases、wiki 等非 git 对象数据。

4. Google Drive 备份语义：
- 仅保留文件内容与目录结构。
- 不保留权限、评论等扩展属性。

5. 恢复范围：
- 当前不实现自动恢复功能（无 `restore` 子命令）。
- 目标是可手动解包：`payload.tar.zst(.enc)` 解密后 untar 得到 bare git repo mirror。

## 12. 结论

整体方案可行，且你的几个关键方向是正确的：
- “单元化 + O(1) 文件 + 可选加密 + 增量跳过”非常适合离线介质备份。
- 先做 GitHub 协议是低风险、反馈快的起点。

建议先锁定上面第 10 节的 5 个决策点，然后就可以进入 GitHub MVP 实现。
